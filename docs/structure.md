# 構造
NerconeOSの構造は大きく分けて3つになります。
- `L0` NerKernel(カーネル)層: スケジューラなどの基本的で重要な機能と、サブモジュール(NKMod)や機能拡張(NKExt)を読み込むためのローダーを含みます。
- `L1` NerKernelモジュール/機能拡張: カーネルの機能を拡張し、より多くの動作を追加するための追加モジュール/機能拡張です。メインモジュールには最低限の機能しかないため、日常利用するにはこれらが欠かせません。
- `L2` サンドボックス層: NerKernel/NKMod/NKExt以外の全てのソフトウェアはこの層で動作します。

## サンドボックス化について
前述の通り、NerconeOSは全てのソフトウェアがサンドボックス上で動作する構造になっています。
また、サンドボックスはそれぞれのソフトウェアごとに隔離され、必要な時に許可されている場所にのみアクセスが可能です。
デスクトップ環境や設定パネルなどの一部のソフトウェアは既定でフルアクセスを持っていますが、基本的にはサンドボックス内へのアクセスのみが許可され、ソフトウェアは必要に応じて追加の権限を要求できます。

これはmacOSの高度なセキュリティ構造を参考にしていますが、NerconeOSではmacOSと違い**全ての**ソフトウェアがサンドボックス上で動作するため、より頑丈で強力になっています。

## バイナリについて
NerconeOSのバイナリは独自のものを採用しています。これは、NerconeOS/NerKernel独自の機能/特徴に対応するためです。

ただし、GCC、ClangやrustcがNerconeOSに対応するとはとても思えない上に、NerconeOS向けのソフトウェアを作る人はほとんどいないでしょうから、Linux/Android/Windows向けのソフトウェアを簡単に実行可能にしています。
仕組みは簡単で、まずLinuxとの互換性を確保するためにLRFN(Linux Runtime for NerconeOS)をサンドボックス上で動かします。
そうすると当然Linuxバイナリも動かせますし、AndroidエミュレータやWineなども動かせます。
これによりNerconeOS向けではないソフトウェアへの互換性も確保しています。

また、WRFN(Windows Runtime for NerconeOS)で使用されるWineは、SodaやProtonなどの派生がとても多いことでも知られます。
特に最近は「このゲームにはこの派生がいい」という情報も多く出回っています。
そこで、NerconeOSでは設定パネルを使用してそれぞれのプラットフォーム用の互換レイヤーを簡単に追加できるほか、実行ファイルごとに互換レイヤーを切り替えることも可能です。

## 外部デバイス対応について
NerconeOSはmacOSと同じように**カメラやマイクなどの外部デバイスはシステム側が処理します**。
つまり、ソフトウェアは外部デバイスへのアクセスについて深いことを考えずにも、簡単にアクセスできます。
ただし、もちろんNerKernelが標準でサポートしていないデバイスもあるでしょう。
その場合、[カーネル機能拡張](nerkernel/nkext.md)を使用すれば解決できますが、万が一機能拡張に脆弱性があった場合は大変です。
そこで[制限された機能拡張](nerkernel/nkext/limited-kext.md)を使用することで、その機能拡張が必要な機能だけを使用できるようになります。これにより、被害が最小限になります。

また、USB Bluetoothデバイスなど、ソフトウェアからでも使用可能なAPIのみでドライバーを実装可能な場合、L1ではなくL2に配置すれば、より被害を減らすことができます。

例えばUSB BluetoothデバイスをNerconeOS SDKを使用して実装する場合、手順は次のようになります。
- [DriverKit](nerconeos-sdk/driverkit.md)を使用しソフトウェアをドライバーとしてNerconeOSに登録する。登録後、NerconeOSはユーザーに詳細情報を提示し、許可された場合にそのソフトウェアへそれらのAPIへの半永久的なアクセス許可を下します。
- [BKDriver](nerconeos-sdk/bluetoothkit/bkdriver.md)構造体を作成します。
    - それぞれのメソッドで、[USBKit](nerconeos-sdk/usbkit.md)を使用し、実際のUSBデバイスにアクセスし、適切なデータを返します。
- [BluetoothKit](nerconeos-sdk/bluetoothkit.md)を使用し、自身のBKDriver構造体をBluetoothKitのドライバリストに登録する。登録が承認されれば、今後のBluetoothデバイスの探索時やペアリング時、通信時などにBluetoothKitがBKDriverの適切なメソッドを呼び出します。
